# Project Agent Guide

这个仓库已经从多时间线互动叙事方向，收口为一个 assistant-centric 的个人 AI 助手项目。

默认产品定位：
- `worldbook` 是背景设定层，不再是产品主轴
- `character card` 是助手人格来源
- `narration` 是低频表达层，不是独立导演视角产品
- 长期记忆、RAG、未来 KG 共同组成 assistant-centric 记忆系统
- `session` 保留为对话片段 / 快照兼容层，不再作为多 timeline 产品主心智

## 默认工程方向

后续改动优先级默认遵循：
1. assistant 主链路可用
2. 数据归属正确
3. 兼容层稳定
4. 回归可重复
5. 再做更复杂的能力扩展

如果没有明确需求，不要重新发散回多 session / 多 timeline 产品方向。

## 前端任务强制流程

当你在这个仓库里负责前端任务时，必须按下面流程工作。

### 1. 先用 `frontend-skill`

- 先定义页面的 visual thesis、信息层级、核心交互方向，再开始改代码。
- 避免 generic AI 风格页面，避免无意义的卡片堆砌。
- 保证首屏重点明确、留白克制、层次清晰。

### 2. 再用 `planner`

- 在动手实现前，先给出一个简洁执行计划。
- 计划里至少包含：目标、要改的文件、组件拆分、响应式策略、验收标准。
- 对复杂任务，先计划再编码，不要直接大改。

### 3. 实现阶段遵循 `frontend-responsive-ui`

- 所有页面和组件都必须兼顾 mobile / tablet / desktop。
- 优先保证布局稳定、断点清晰、间距一致、文本不溢出、按钮可点击区域合理。
- 不要只做桌面版效果。

### 4. 需要查项目或第三方资料时

- 先用 `read-github` 读取当前仓库相关实现、README、约定和已有模式。
- 需要框架或库的最新用法时，使用 `ctx7old` 查文档后再实现。
- 优先复用项目现有模式，不要随意引入新的架构风格。

### 5. 完成后必须做 `ui-audit`

- 检查视觉一致性、可访问性、对齐、间距、状态反馈、交互完整性。
- 检查是否存在明显的 UX 问题、低级错位、不可读文本、按钮状态缺失、hover/focus 缺失。
- 发现问题先修，再交付。

### 6. 最后做 `typography-audit`

- 检查字号层级、字重、行高、段落间距、标题和正文关系。
- 保证排版节奏统一，避免字体大小混乱、标题太密、正文太挤、视觉重心漂移。

### 7. 交付要求

- 最终输出时，先简要说明做了什么。
- 列出修改的文件。
- 说明如何验证。
- 若有未完成项或风险，明确写出，不要假装已经完成。

## 通用约束

- 优先沿用现有设计系统、组件风格、目录结构和工程约定。
- 除非必要，不新增依赖。
- 不为了“看起来炫”而牺牲可读性、性能和可维护性。
- 如果任务不明确，先基于现有代码和上下文做最合理方案，不要停在空泛建议上。
- 复杂任务默认先计划，再实施，再自检。

## 数据与兼容层约束

- assistant 是主入口；session 是兼容快照层。
- 不要粗暴删除旧逻辑，优先通过兼容层逐步迁移。
- 改动 assistant、memory、session 归属时，要明确评估兼容性。
- 回归脚本、测试 fixture、临时数据都要可清理、可复用，避免持续污染 `game-data`。

## 协作要求

- 在做较大改动前，先说明改动目标和第一步。
- 优先给出小步可验证的结果。
- 如果发现仓库现状与 README、guidance 或接口文档不一致，优先修正文档或指出差异。
- 如果有多种实现路径，默认选择更稳、更便于后续维护的一条。
