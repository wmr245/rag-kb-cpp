# 提升召回准确率的优化说明

这份文档用于说明当前项目为了提升 RAG 召回准确率，已经做了哪些工程和算法侧工作，便于面试讲解、后续协作和继续迭代。

## 目标

当前优化目标不是一次性替换整条检索链路，而是在保留现有 `pgvector + 规则重排` 主链路的前提下，用低风险、可验证的方式逐步提高召回准确率、结构定位能力和结果可解释性。

## 已完成工作

### 1. 提升文档切块质量

为了让进入向量库的数据更干净、更有结构，已经对文档解析和切块做了增强：

- 对 `txt / md / pdf / docx` 做了差异化读取。
- Markdown 支持标题感知切块，不再只是纯长度切分。
- PDF 按页提取，并保留 `page` 信息，便于 citation 对齐。
- 切块前做文本归一化、智能并行合并和长段落硬切分，减少噪音和无意义断点。

这一步的价值是：先提升原始 chunk 质量，再谈后续召回和排序优化。

### 2. 给 chunk 补齐结构化 metadata

为了让检索阶段不仅依赖纯文本和向量相似度，已经把下面这些 metadata 纳入 `chunks` 表：

- `heading`
- `section_path`
- `chunk_type`
- `source_type`

其中：

- `heading` 用于表示当前 chunk 的主标题。
- `section_path` 用于表示从上层标题到当前标题的层级路径，例如 `RAG Smoke Test > 系统目标`。
- `chunk_type` 用于区分 `section / paragraph / page` 等类型。
- `source_type` 用于区分 `md / pdf / txt / docx` 等来源。

这一步的价值是：让后续排序逻辑可以利用文档结构，而不是只看正文文本。

### 3. 为历史知识库做 metadata 回填

为了避免只有新文档带 metadata、旧文档仍然缺失结构信息，已经补了一个批量回填脚本，对历史 `ready` 文档重新走 metadata-aware ingest：

- 脚本位置：`python-ai/app/scripts/backfill_chunk_metadata.py`
- 回填方式：按现有 `source_path` 重读文档、重新切块、重新 embedding、重写 chunk 行
- 结果：当前知识库内 chunk metadata 覆盖率已经补齐到全量

这一步的价值是：让新的召回优化可以覆盖整个知识库，而不是只对新上传文档生效。

### 4. 在 rerank 中使用 metadata

为了让召回结果更符合用户问题的结构意图，已经把 metadata 接入到检索重排逻辑中：

- 使用 `heading` 和 `section_path` 参与关键词重叠打分。
- 对标题、章节、小节、部分等结构定位类问题做额外加权。
- 对定义类问题，优先提升更像“定义段”的 chunk。
- 对步骤类、流程类问题，优先提升更像“步骤段”的 chunk。
- 对 Markdown 结构化文档，在结构查询场景下给予适度 bias。

这一步的价值是：即使向量分数接近，也能让更符合“问题意图”的 chunk 往前排。

### 5. 引入混合检索

为了减少“只靠向量相似度时，字面词命中不够稳定”的问题，已经补上了第二条关键词召回通路：

- 保留原有 `pgvector` 向量召回作为主召回链路。
- 新增基于 `pg_trgm` 的关键词候选召回，覆盖 `title / heading / section_path / text`。
- 对向量候选和关键词候选做去重合并，再进入统一 rerank。
- 对“同时被两条通路命中”的 chunk 给予轻量 bonus，提高稳健性。

这一步的价值是：兼顾语义相似与字面匹配，让结构定位、标题命中、术语命中这类查询更稳定。

### 6. 针对结构定位问题扩大候选集

为了减少“正确 chunk 在粗召回阶段就被截断”的情况，已经针对结构型、流程型问题适度放大 `candidate_k`：

- 普通问题维持当前候选规模。
- 标题定位、章节定位、流程步骤类问题扩大候选集上限。

这一步的价值是：提高结构化查询场景下的 Recall@K，为 rerank 留出更多可选候选。

### 7. 补强验证和可观测性

为了确保优化不是“感觉更好”，而是能被验证：

- 新增/增强了端到端自测脚本 `scripts/smoke_test.py`
- 自测结果会打印命中 chunk 的 `heading / sectionPath / chunkType / sourceType`
- 可以直接观察召回结果是否在结构上更合理

这一步的价值是：后续每次改检索策略，都能快速看出结果是否退化。

## 当前收益

完成以上工作后，系统在下面这些问题上会更稳：

- 问“第几个标题/小节/章节是什么”
- 问“某个概念在哪个部分提到”
- 问“某个流程/步骤怎么做”
- 问“文档主要讲了什么”时优先命中更像章节概述的 chunk

同时，返回结果现在也更容易解释：不只是返回一段文本，还能说明它来自哪个标题路径、属于什么 chunk 类型。

## 为什么这条路径更稳

这次没有直接引入重型框架、BM25、外部 reranker 或复杂 Agent，而是优先做了：

1. 先把 chunk 数据质量做扎实。
2. 再把 metadata 补齐到全量知识库。
3. 再让 rerank 真正利用这些结构信息。

这样做的好处是：

- 风险低
- 容易验证
- 对面试讲解友好
- 也为后续混合检索和轻量 reranker 打下了基础

## 下一步建议

如果继续往“更高召回准确率”方向推进，推荐顺序是：

1. 建立离线评测集，对召回准确率做量化比较
2. 增加基于 query 类型的回归样例和对拍脚本
3. 在有评测基线后，再考虑接入更轻量的 reranker
4. 继续细化 query intent 和 citation 对齐策略
