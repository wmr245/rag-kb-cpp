# 2026-03-21 Next Prompt

你现在接手的是一个已经完成基础工程化收口的 RAG 项目，不要从零开始分析。先基于现状继续推进，而不是重复搭框架。

当前系统已经具备：
- 文档上传、异步入库、混合检索
- chunk metadata（heading / sectionPath / chunkType / sourceType）
- metadata-aware rerank
- 拒答策略与 citation 对齐
- smoke test、分层评测集、报告对比、baseline 机制
- 默认 embedding 运行配置已统一为 `text-embedding-v4 + 1536`

你下一步的主要目标，不是继续零散调规则，而是补齐“文档级路由”这一层，让默认查询不再依赖用户显式提供 `docId`。

请按下面顺序继续：

## 1. 实现文档级总表 / 路由层

目标：
- 增加一个文档级 catalog 能力，用于保存每篇文档的标题、摘要、关键词、来源类型、更新时间等信息
- 当用户不传 `docScope` 时，先在文档级 catalog 中做一次候选召回
- 从 catalog 中选出 top 3 到 top 5 个疑似相关文档
- 再把这些候选文档的 `docId` 送入现有 chunk 检索主链路
- 如果用户显式传了 `docScope`，则保持当前行为，不走自动路由

要求：
- 路由优先采用“检索优先”的方式，不要直接让大模型从全库猜文档
- 可结合标题、摘要、关键词、文档级 embedding、trigram/BM25 等方式
- 如果需要 LLM 参与，也只允许它在已召回的少量候选文档里做 rerank，而不是直接全库判断

## 2. 把文档级路由纳入评测

需要新增评测 case，至少覆盖：
- 默认不传 `docScope` 的普通问答
- 结构定位类问题
- 多文档语义容易混淆的问题
- 明确应拒答的问题

目标不是只看能不能答，而是确认：
- 文档级候选是否合理
- 最终命中的 citation 是否仍然对齐
- 拒答没有明显退化

## 3. 做完后给出批判性评估

实现完成后，不要只汇报“功能做完了”，还要明确判断：
- 文档级路由是否真正改善了默认查询体验
- 是否降低了全库检索噪声
- 哪些 query 仍然表现一般
- 还剩哪些 RAG 层面的关键短板

## 4. 当前阶段的结论要延续

下次推进时请默认采用下面这个判断，不要重新摇摆：

- 当前 RAG 已经超过 Demo 阶段，底座具备继续向产品形态演进的条件
- 但 RAG 本身还没有做到“没有明显进步空间”
- 在进入 Agent 之前，最值得补的一步仍然是“文档级路由 / 分层检索”
- 行业更成熟的系统也不是让 Agent 取代 RAG，而是把 Agent 建在更稳的检索、rerank、证据控制和评测体系之上
- 因此，最稳妥路线是：先完成文档级路由，再开始做一个克制的单 Agent，而不是直接上复杂多 Agent

## 5. Agent 的启动边界

如果文档级路由完成且评测稳定，可以开始做第一版 Agent，但范围要控制住：
- 先做单 Agent
- 只负责查询范围判定、必要时问题拆解、调用现有检索与回答链路
- 不要一上来做多 Agent 编排
- Agent 的效果必须继续纳入现有 eval / compare / baseline 体系

总原则：
- 继续沿用现有工程化方法
- 优先做“可验证提升”的改动
- 不要为了看起来更智能而牺牲可控性、引用对齐和拒答稳定性
